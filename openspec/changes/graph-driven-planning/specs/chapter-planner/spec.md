## MODIFIED Requirements

### Requirement: Build planning prompt from algorithmic plan + knowledge graph
The system SHALL construct a planning prompt that presents pre-computed algorithmic constraints (community-based Part groupings and topological chapter ordering) to Claude. When a knowledge graph is available, the prompt SHALL include: the AlgorithmicPlan (communities, topo order, centrality scores), extracted Concept/Workflow/DataModel node summaries, and project metadata. Claude's role is to name Parts, write chapter titles/subtitles/outlines, and optionally reorder within non-constrained positions — but NOT to change the community grouping or violate DEPENDS_ON ordering.

#### Scenario: Prompt with algorithmic plan and knowledge graph
- **WHEN** knowledge-graph.json exists and `compute_algorithmic_plan()` has produced 3 communities with topo-sorted nodes
- **THEN** the prompt presents each community as a "Part candidate" with its ordered node list and centrality scores, instructs Claude to "assign compelling Part titles, Chapter titles, and detailed outlines while preserving the prerequisite ordering"

#### Scenario: Prompt without knowledge graph (fallback)
- **WHEN** no knowledge-graph.json exists and analyze phase failed
- **THEN** the system falls back to the original directory-tree-based prompt via `_generate_fallback_skeleton()`

### Requirement: Call ClaudeClient for chapter planning with constrained structure
The system SHALL use `ClaudeClient(cwd=repo_path).run(prompt, max_turns=30)` to invoke Claude. The prompt SHALL be generated by the graph-driven planner with algorithmic constraints when a knowledge graph is available.

#### Scenario: Graph-driven planning call with constraints
- **WHEN** knowledge graph exists and `plan_from_graph()` is invoked
- **THEN** Claude receives a prompt with pre-computed Part boundaries and chapter ordering, and returns a `parts[].chapters[]` JSON that respects the algorithmic constraints while adding creative naming and outlines

### Requirement: chapter-planner reads chapter-outline.json in run loop
During `pyharness run` Plan phase, when `chapter-outline.json` exists, the planner SHALL read it to access `uncovered_nodes` and `kg_coverage` information, using this to prioritize which chapters need expansion or which new chapters should be added.

#### Scenario: Re-planning with existing outline
- **WHEN** `pyharness run` enters Plan phase and `chapter-outline.json` exists with 2 uncovered nodes
- **THEN** the planner includes those uncovered nodes in the prompt context, suggesting Claude consider adding coverage for them
